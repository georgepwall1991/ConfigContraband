# Analyzer Health

This file tracks the current ConfigContraband analyzer surface and the next hardening work that is still worth doing. It should stay practical: scores drive priority, notes describe shipped behavior, and gaps should be specific enough to turn into a focused PR.

Last refreshed: 2026-07-06
Package version: `0.5.12`
Base audited commit: `f5beb02`

## Scoring Rubric

Each category is scored from 1 to 5. Higher is better except `Importance`, where higher means the rule is more valuable to harden.

| Category | Weight | 1 | 3 | 5 |
|---|---:|---|---|---|
| Importance | 25% | Cosmetic or rare case | Common correctness issue | High-impact production failure class |
| Precision | 25% | Broad or likely false positives | Conservative for common cases | Strong safe/unsafe boundaries |
| Test Depth | 20% | Happy path only | Main positive and negative cases | Edge cases, aliases, duplicates, and regressions |
| Fix Safety | 15% | Unsafe or missing where expected | Fix exists but needs edge coverage, or manual rule is documented | Fix is deterministic, narrow, and regression-tested |
| Documentation | 10% | Undocumented behavior | README/catalog summary exists | Safe/risky shapes and no-fix rationale are documented |
| Release Readiness | 5% | Blocks shipping | Minor gaps only | CI/verifier/docs are current |

Overall score:

```text
Importance * 0.25 + Precision * 0.25 + TestDepth * 0.20 + FixSafety * 0.15 + Documentation * 0.10 + ReleaseReadiness * 0.05
```

Priority means "next investment priority", not diagnostic severity:

| Priority | Meaning |
|---|---|
| P1 | Best next hardening target. |
| P2 | Worth improving after P1 work or when touching the shared helper. |
| P3 | Healthy enough; monitor for real-world gaps. |

## Current Posture

The analyzer has a compact, coherent rule set: seven diagnostics (`CFG001`-`CFG007`), four code-fix families, `BindConfiguration(...)`, explicit `Bind(GetSection(...))`, direct `Configure<T>(GetSection(...))` section/key checks, required-key warnings gated on DataAnnotations validation, strict `ErrorOnUnknownConfiguration` unknown-key warnings, fluent-chain validation checks before and after binding calls, `buildTransitive` appsettings discovery, README rule documentation, changelog/version metadata, and CI that restores, builds, tests, packs, uploads test results, and uploads packages. Since the last audited analyzer baseline, only dependency/workflow and pi-extension evidence updates landed; analyzer source behavior did not change, though the 2026-06-30 evidence-based audit described below corrected several stale rule-documentation claims. The local release-grade verifier now passes again: `dotnet test ConfigContraband.slnx --configuration Release`, `dotnet format ConfigContraband.slnx --verify-no-changes --no-restore --verbosity minimal`, and `git diff --check` all passed on 2026-06-30. Release Readiness is restored to `5` across every rule, so release-readiness itself returns to monitor mode rather than infrastructure-restoration mode — but the same 2026-06-30 audit (below) found real precision/test-depth gaps that moved `CFG001`, `CFG002`, `CFG004`, and `CFG005` out of monitor mode on those specific dimensions. `CFG001`'s chained-`GetSection(...)`-off-a-stored-section gap was fixed on 2026-07-01, `CFG002`'s unpinned collection-default behavior was pinned with regression tests the same day, and `CFG004`/`CFG005`'s shared type-level-attribute-inheritance gap was fixed the same day too (see Improvement Changelog), returning all four rules to monitor mode. Every rule reached P3 / monitor by 2026-07-01, and a 2026-07-06 monitor sweep (Improvement Changelog iteration 85) found nothing new — but a same-day, deeper eight-reviewer evidence audit (iteration 86, described in the next paragraph) then found five concrete new gaps, moving `CFG001`, `CFG002`, `CFG003`, `CFG006`, and `CFG007` out of monitor mode and into the Current Shortlist as P1/P2 hardening targets. `CFG004` and `CFG005` remain P3 / monitor. `CFG001`'s conditional-access (`?.`) chain gap, `CFG007`'s tuple-deconstruction escape gap, and `CFG002`'s primary-constructor-parameter-default gap were all fixed the same day (see Improvement Changelog), returning all three to P3 / monitor mode. A later same-day eight-reviewer evidence audit (iteration 91) then found four *new* concrete precision gaps that all prior sweeps and audits had missed — `CFG001`'s section-typo code fix silently drops path segments when a chained non-root `GetSection(...)` literal itself contains a colon (unsafe fix), `CFG002` never fires for a property annotated with a user-defined `RequiredAttribute` *subclass* the runtime validator does enforce (false negative), `CFG003` emits a *false positive* on a local-variable builder whose `ValidateOnStart()` sits after an intervening unrelated statement (distinct from the already-known parameter-typed split-statement false negative), and `CFG007` fires a false Warning when a reset delegate escapes as a call argument that is then invoked (false positive). All four were empirically reproduced and adversarially confirmed against source, and each took a one-dimension score drop (see Improvement Changelog iteration 91 and the Health Baseline). `CFG001`, `CFG002`, and `CFG007` moved out of monitor mode into the Current Shortlist; `CFG003` was already a P2 shortlist item (the iteration-86 parameter-typed split-statement false negative) and this additional, distinct false positive escalated it to P1. The same audit also established that `CFG005` shares `CFG006`'s `IsPotentialNestedObject` `TypeKind.Class` filter, so a nested struct/record-struct type carrying validation attributes is silently skipped by `CFG005` too — that shared struct false negative was previously documented only against `CFG006`, so `CFG005` also drops one dimension and moves to P2 (one shared-helper fix closes both), and the audit fixed two stale internal Current-Shortlist cross-references in the CFG003/CFG006 Known Gaps. `CFG004`, CI, README, `AnalyzerReleases`, package metadata, and the Health Baseline score arithmetic were all independently re-verified accurate with no changes; no analyzer, code-fix, or test source changed in this audit. The iteration-91 `CFG003` P1 (forward split-local false positive) was fixed on 2026-07-06 (iteration 92), which also removed a latent CFG004 false positive on the shared scanner and restored `CFG003` to P2; the `CFG001` colon-delimited chained-literal unsafe fix was fixed on 2026-07-06 (iteration 93), restoring `CFG001` to P3/monitor; the `CFG007` reset-delegate-as-argument false positive was fixed on 2026-07-06 (iteration 94), restoring `CFG007` to P3/monitor; the `CFG006`/`CFG005` shared struct nested-object false negative was fixed on 2026-07-06 (iteration 95), restoring both to P3/monitor; the next iterations should harden the remaining P2 shortlist target `CFG002` (user-defined `RequiredAttribute` subclass false negative), plus the still-open `CFG003` parameter-typed split-statement false negative and its newly-characterized backward-scan mirror — rather than a monitor sweep.

A later, deeper evidence-based audit that same day (2026-06-30, eight parallel reviewers, one per rule plus a cross-cutting infra/docs pass) went past that metadata-freshness sweep and re-verified every Rule Notes claim against current source and tests, including ad hoc empirical probes reverted before completion. It found two concrete, evidence-backed precision gaps that prior monitor sweeps had not surfaced: `CFG001` does not ignore a `GetSection(...)` call chained off a previously stored `IConfigurationSection` variable, instead checking it against the root namespace, and `CFG004`/`CFG005` both miss a type-level `ValidationAttribute` declared only on a base options class because both rules detect validation attributes through the same `ContainsValidationAttributes` helper, which does not reuse the inheritance-aware walk `CFG002` already relies on for the same shape (`GetNestedValidationCandidates` calls the identical helper for `CFG005`'s nested-object case). No analyzer, code-fix, or test source changed in this later sweep either, but its narrower, rule-by-rule evidence supersedes the "unchanged" read above for `CFG001`, `CFG002`, `CFG004`, and `CFG005` specifically: those four rules' Test Depth (and CFG001/CFG004's Precision — CFG005's Precision was already `4`) scores move down to reflect the gaps, and `CFG001`/`CFG004`/`CFG005` move from monitor mode to concrete P1/P2 hardening targets in the Current Shortlist — the `CFG004`/`CFG005` fix is the same shared-helper change, so one PR resolves both. `CFG003`, `CFG007`, CI, AnalyzerReleases, and package metadata were all independently verified accurate with no changes, while `README.md`'s `CFG004` section was corrected alongside this doc to match.

A follow-up evidence-based audit on 2026-07-06 (eight parallel reviewers, one per rule plus a cross-cutting infra/docs pass) re-checked the fully-monitor-mode posture reached on 2026-07-01 and found five new concrete gaps that prior sweeps had not surfaced. `CFG001` silently skips the entire section-existence check for any registration that uses `?.` (conditional access) anywhere before or between `GetSection`/`GetRequiredSection` calls — even a single non-chained `configuration?.GetSection("Strpie")` — because `UnwrapForSectionChainResolution` only unwraps parenthesized and null-forgiving (`!`) wrappers, not conditional-access expressions, so `TryGetConfigurationSectionPath`'s `is not InvocationExpressionSyntax` guard rejects the whole expression before ever resolving a path. `CFG002` reports a false positive for a `[Required]` property whose primary-constructor (C# 12) parameter default provably satisfies the attribute, because `ConstructorParameterDefinitelyReachesProperty`'s proof only recognizes `ConstructorDeclarationSyntax` bodies (ordinary constructors and positional records), not a primary constructor's parameter list living on the `TypeDeclarationSyntax`, and the initializer fallback proof also fails because the property initializer is an identifier reference to the parameter rather than a constant or object-creation expression. `CFG003` still tracks a single fluent chain on a parameter-typed builder correctly, but its same-block split-statement local-chain scanner requires an `ILocalSymbol` receiver, so a parameter-typed builder's bind and validation calls split across separate statements go untracked. `CFG006` excludes struct- and record-struct-typed nested options properties from recursive unknown-key analysis because `IsPotentialNestedObject` filters on `TypeKind.Class`, even though the real `ConfigurationBinder` does bind and recurse into struct-typed properties. `CFG007` cannot see a tuple-deconstruction assignment to `ErrorOnUnknownConfiguration` (for example `(options.ErrorOnUnknownConfiguration, options.BindNonPublicProperties) = (true, false);`) because the escape-analysis proof only inspects an `AssignmentExpressionSyntax` whose `.Left` is directly the target member access, not a member access buried inside a `TupleExpressionSyntax`, so strict binding that is genuinely enabled at runtime is missed and the diagnostic silently falls back to the softer `CFG006` Info. `CFG004`, `CFG005`, CI, package metadata, `AnalyzerReleases`, README, and the Health Baseline table's own score arithmetic were all independently re-verified accurate with no changes. These five findings move `CFG001`, `CFG002`, `CFG003`, `CFG006`, and `CFG007` out of monitor mode into the Current Shortlist; no analyzer, code-fix, or test source changed in this audit.

## Selection Policy

Pick one focused rule per hardening batch unless two rules share the same helper boundary. Prefer:

1. Warning rules before Info rules when scores are close.
2. High `Importance` with low `Precision`, `Test Depth`, or `Fix Safety`.
3. Fixable rules with unproven code fixes before manual-only documentation cleanup.
4. Small changes that can be verified with targeted tests.

Do not raise severity, rename analyzer IDs, or broaden diagnostics unless tests prove the behavior is safe.

## Rule Notes

### CFG001 Missing Configuration Section

Reports when a supported options binding references a string-literal section path that does not exist in visible `appsettings.json` or `appsettings.*.json` files. Nested section paths are matched segment by segment. The code fix replaces the literal with the corrected section path, or the corrected leaf when the code uses chained `GetSection(...)` calls.

Current behavior:

- Checks top-level and nested section paths across all visible `appsettings.json` and dot-qualified `appsettings.*.json` additional files for `BindConfiguration(...)`, `Bind(GetSection(...))`, `Bind(GetRequiredSection(...))`, and direct `Configure<T>(GetSection(...))`.
- Parses `//` and `/* ... */` comments, JSON string escapes, and colon-delimited JSON keys in appsettings files before resolving section paths.
- Traverses duplicate JSON object members when resolving section existence and suggestions.
- Preserves regular, verbatim, and raw string literal style when applying section typo fixes, and falls back to an escaped normal string when a raw replacement would contain a decoded line break.
- When the reported section literal is a chained non-root `GetSection(...)` argument, the fix rewrites only that literal's leaf and preserves any leading colon segments the literal itself carries (`.GetSection("Features").GetSection("Sub:Strpie")` fixes to `.GetSection("Sub:Stripe")`, not `.GetSection("Stripe")`); the fix is suppressed when the anchored section expression is not a plain string literal whose segments can be reproduced safely.
- Ignores non-constant, empty, whitespace-only, root configuration, a directly-passed stored `IConfigurationSection` value (no literal path argument to check in that shape), and a further `GetSection(...)`/`GetRequiredSection(...)` call chained off any stored/received `IConfigurationSection`-typed receiver (a local, parameter, or other section-typed expression, including a nullable-annotated `IConfigurationSection?`) rather than checking the chained literal against the root namespace.
- Resolves chained `GetSection(...)` calls off a genuine root-configuration chain even when the inner call is wrapped in parentheses or a null-forgiving `!` (for example `(configuration.GetSection("Features")).GetSection("Strpie")` or `configuration.GetSection("Features")!.GetSection("Strpie")`), still proving and checking the full constant path.
- Resolves a genuine root-configuration chain through conditional access (`?.`) anywhere in the chain — a non-chained call (`configuration?.GetSection("Strpie")`), a chain with `?.` right after the root (`configuration?.GetSection("Features").GetSection("Strpie")`), `?.` before a further chained call (`configuration.GetSection("Features")?.GetSection("Strpie")`), or a null-forgiving `!` immediately after a `?.`-bound call followed by a further chained call (`configuration?.GetSection("Features")!.GetSection("Strpie")`) — still proving and checking the full constant path instead of silently skipping the registration. Fixed 2026-07-06 (see Improvement Changelog).

Known gaps:

- Treats visible appsettings files as one searchable set and does not model configuration-provider ordering.
- Does not infer dynamic section names.
- Chaining a further `GetSection("X")` call off a stored `IConfigurationSection`-typed variable or parameter is intentionally ignored rather than resolved relative to the stored section's own (unknown, non-constant) path, so a real typo under that nested path stays undetected. This is the conservative side of the 2026-07-01 fix (see Improvement Changelog): false-positives/false-negatives from checking the chained literal against the wrong namespace are closed, but the analyzer does not attempt to track the stored section's origin to resolve the full path and re-enable the check.
- **Fixed 2026-07-06 (iteration 93) — unsafe fix**: when a chained non-root `GetSection(...)` literal itself contains a colon (e.g. `.GetSection("Features").GetSection("Sub:Strpie")`), the recursion in `TryGetConfigurationSectionPath` reports the full path (`Features:Sub:Strpie`) but records the anchored literal as `"Sub:Strpie"` with `sectionExpressionContainsFullPath=false`. `AnalyzeConfigurationSection` previously stored only the corrected *leaf* (`"Stripe"`) as the replacement, so `ReplaceSectionAsync` overwrote the entire `"Sub:Strpie"` literal with `"Stripe"` and silently dropped the `Sub:` segment, contradicting the fix's own label. Fixed by `TryBuildChainedLeafReplacement`: the replacement is now `ReplaceSectionLeaf(anchoredLiteralValue, suggestion)`, preserving the leading segments (`"Sub:Stripe"`), and the fix is suppressed (no replacement property, so no auto-fix is offered while the diagnostic and "Did you mean" message remain) when the anchored expression is not a plain string literal whose leading segments can be reproduced safely. Regression test: `Cfg001_fix_preserves_colon_segments_in_chained_get_section_leaf_literal`. Residual: a chained non-root section literal supplied through a `const`/non-literal expression whose value spans multiple segments now offers no auto-fix rather than risk a segment-dropping edit — a deliberate, documented conservative trade.

### CFG002 Missing Required Configuration Key

Reports when a supported binding with a visible DataAnnotations validation path declares a `[Required]` property whose key is missing from every visible appsettings section and whose compile-time default cannot satisfy the attribute.

Current behavior:

- Runs only when DataAnnotations validation is provable: `OptionsBuilder<TOptions>` chains with `ValidateDataAnnotations()`, or direct `Configure<TOptions>(GetSection(...))` calls whose same top-level block registers matching `AddOptions<TOptions>().ValidateDataAnnotations()` (including default-name, empty-string-name, and configure-all shapes).
- Reports `[Required]` reference, string, and nullable value properties; C# `required` members and `[Required]` on non-nullable value types stay quiet because they cannot fail the runtime null check.
- Stays quiet when a compile-time default provably satisfies `RequiredAttribute` and `[Required]` is the property's only validation constraint (other validators on the property, type-level validation attributes anywhere in the inheritance chain, and `IValidatableObject` on the options type keep the key required because they also validate the default; the recursive walk inspects every public-getter property the runtime validator sees, not just bindable ones): compile-time constants resolved by value (literals, signed numerics such as `= -1`, `const` fields, `nameof`; constants flowing through user-defined conversions stay conservative), any non-null constant string when `AllowEmptyStrings = true` (including constructed strings, which are non-null), object-creation initializers on non-string properties (including target-typed `new()` on nullable value types, which constructs the underlying value; creations flowing through user-defined conversions stay conservative), a property initializer that directly references a primary-constructor (C# 12) parameter with a satisfying explicit default value (judged by the parameter's own default, the same way as any other satisfying initializer expression, with the same user-defined-conversion guard; fixed 2026-07-06, see Improvement Changelog), and single-constructor parameter defaults that provably reach the property (constructor-bound properties whose parameter default is not satisfying still fall back to the initializer proof when no declared constructor overwrites them; every proof requires an auto-implemented getter because validation reads the getter) — positional record parameters, or a constructor body containing nothing but simple parameter-or-literal assignments to fields or non-overridable auto-implemented properties including `Property = parameter;`, with no other writes to the property, no name hiding in the type chain, and no shadowing parameter or local (custom setters and virtual members on assigned targets could mutate the required property, so they invalidate the proof). Empty or whitespace-only strings, `null!`/`default` initializers, explicit parameterless `Nullable<T>` construction resolved through the semantic model (`new int?()` or any alias — an empty `Nullable<T>` boxes to null regardless of the declared property type), constructed strings such as `new string(' ', 3)` on any property type, constructor defaults that are never assigned to the property or whose constructor body contains helper calls, parameter reassignments, or other statements, satisfying initializers whose runtime-selected constructor chain (the binding constructor plus implicitly chained base constructors; zero-argument `: this()`/`: base()` chains are followed, while argument-carrying chains and syntaxless metadata base constructors stay conservative) could overwrite the property, recursive-validation parents whose walked default graph contains anything besides provably satisfied `[Required]` members (other validation attributes, `IValidatableObject`, initializer expressions, constructor arguments, or polymorphic creations stay conservative; collection defaults too), constructor-bound properties whose name is hidden anywhere in the type chain, and non-constant initializers such as method calls keep reporting because the analyzer cannot prove the runtime value passes validation.
- Merges all visible `appsettings.json` and dot-qualified `appsettings.*.json` files with union semantics, so environment-specific override files satisfy required keys.
- Uses `[ConfigurationKeyName]` aliases for property-bound options and constructor parameter keys for constructor-bound properties, and reports the configured key name in the diagnostic message.
- Recurses into nested objects and collection elements only where Options validation walks: `[ValidateObjectMembers]` and `[ValidateEnumeratedItems]`, including provably initialized nested objects whose section is missing entirely (null members and empty collection defaults are skipped, matching runtime validation; unprovable defaults keep the parent required without speculative child diagnostics). Dictionary values stay quiet because recursive Options validation attributes do not validate dictionary values.
- Stays quiet when the bound section itself is missing so `CFG001` carries the signal alone.
- Shares its required-key contract with the generated `appsettings.schema.json` `required` array through the same metadata flag.

Known gaps:

- Required values supplied only through environment variables, user secrets, or other non-JSON providers are invisible to the analyzer and remain the documented residual false-positive source.
- **P2 false negative, found 2026-07-06 (iteration 91)**: `IsRequired()` (OptionsMetadata.cs ~623-624) treats a property as required only when an attribute's `ToDisplayString()` exactly equals `System.ComponentModel.DataAnnotations.RequiredAttribute`, so a property annotated with a user-defined `RequiredAttribute` *subclass* (e.g. `class MyRequiredAttribute : RequiredAttribute { }`, no weakening `IsValid` override) yields `IsRequired=false` and CFG002 never fires — even though `Validator.TryValidateObject` enforces the inherited attribute and throws `OptionsValidationException` at runtime when the key is absent. (`HasValidationAttribute`/`HasNonRequiredValidationAttribute` correctly use inheritance-aware `InheritsFrom` checks, but those flags drive CFG004/CFG005, not the missing-key report.) Empirically reproduced 2026-07-06 (medium confidence — user-defined `RequiredAttribute` subclasses are uncommon). See Current Shortlist item 5.
- `[Required]` collection and array properties default-initialized with any array-creation expression (`= []`, `= new[] { ... }`, `= new string[0]`, `= new string[] { ... }`) still report, even though `RequiredAttribute` accepts any non-null non-string value: `InitializerDefinitelySatisfiesRequired` only handles `ObjectCreationExpressionSyntax`/`ImplicitObjectCreationExpressionSyntax`, with no `ArrayCreationExpressionSyntax` or collection-expression branch, so every array-shaped default stays on the conservative reporting path regardless of explicit, implicit, or collection-expression syntax. This narrow widening was reviewed and deliberately declined to keep that boundary closed; revisit only on real-world reports. Non-array collection/dictionary properties default-initialized via object-creation syntax (`= new List<T>()`, `= new()`) already satisfy the existing object-creation-initializer suppression and do not report today — that was not an intentional collection-specific design decision, but the shape is now pinned by regression tests (2026-07-01, see Improvement Changelog) so it can't silently regress unnoticed.
- **Accepted false negative (default-struct field initializers), 2026-07-06 (iteration 95)**: when a non-nullable struct nested-options property is absent from config, the missing-section recursion walks its `default(T)` instance, but the nested required-satisfaction check (`HasRequiredSatisfyingDefault`) still credits the nested struct type's own property/field initializers and parameterless-constructor assignments. C# `default(T)` does *not* run those (only `new T()` does), so a nested struct such as `public struct DatabaseOptions { public DatabaseOptions() {} [Required] public string ConnectionString { get; set; } = "ok"; }` is treated as satisfied even though `default(DatabaseOptions)` leaves `ConnectionString == null` and `Validator.TryValidateObject` fails at runtime. Correctly modelling `default(T)`-vs-`new T()` initializer semantics is a materially larger change than the struct-nesting widening it rides on; this residual is a false negative (the safe side of the rule's prefer-false-negatives bias), on an extraordinarily rare shape (a struct with an explicit parameterless constructor *and* member initializers, nested under an absent non-initialized struct property), and it is strictly narrower than the pre-iteration state where struct nested objects were ignored entirely. Deliberately deferred (Codex review round 5, iteration 95); revisit only on real-world reports.
- No automatic fix because the missing key lives in JSON additional files.

### CFG003 Validation Not On Startup

Reports when an options registration has validation through `ValidateDataAnnotations()` or `Validate(...)` but no `ValidateOnStart()`.

Current behavior:

- Tracks normal fluent chains before and after `BindConfiguration(...)` and `Bind(GetSection(...))`.
- Covers named and unnamed `OptionsBuilder<TOptions>` registrations when validation stays on the same supported builder chain.
- Tracks immediate same-block local `OptionsBuilder<T>` calls, including binding in the initializer or a later local expression statement. For later local binding statements, same-local calls before and after the bind are recognized, as are recognized invocations in the adjacent builder initializer. The *forward* split-statement scan (`AddSubsequentLocalInvocations`) skips inert intervening statements — an unrelated call/assignment (e.g. an interleaved `services.AddSingleton<X>()`) or a local declaration — and keeps collecting same-local calls, stopping when the tracked builder local is reassigned or at a control-flow statement (`if`/`return`/`throw`/loops/etc.) that could keep the later call from running on every path; the *backward* prior-scan still stops at the first unrelated statement (see Known Gaps).
- Counts only the framework `OptionsBuilder<TOptions>.Validate(...)`, `ValidateDataAnnotations()`, and `ValidateOnStart()` APIs as validation signals; custom same-name helpers are ignored by name alone.
- Treats `AddOptionsWithValidateOnStart<TOptions>()` as startup validation, so registrations using the framework helper do not need an extra `ValidateOnStart()` call.
- Offers a fix that appends `ValidateOnStart()` while preserving multiline chain indentation, comments, split locals, and single-line chains.

Known gaps:

- Does not infer non-local builder storage, aliases, reassignment, or broader control flow.
- **Accepted false negative (delegate-capture retarget), 2026-07-06 (iteration 92)**: the forward split-statement scan's `StatementRetargetsLocal` guard recognizes a direct assignment, a tuple-deconstruction target, and a `ref`/`out` argument as retargeting the tracked builder local, but it deliberately does not descend into lambda/local-function bodies (so a deferred assignment inside an un-invoked delegate is correctly *not* treated as an immediate retarget). The dual shape — a local delegate that captures and reassigns the builder and is then *invoked* between the bind and the later validation call (`Action reset = () => optionsBuilder = services.AddOptions<T>(); reset(); optionsBuilder.ValidateOnStart();`) — is not detected, so CFG003/CFG004 can miss a diagnostic there. Correctly distinguishing the invoked case from the never-invoked case (which must stay quiet) requires delegate-capture/invocation dataflow the rule explicitly disclaims, and any simple syntactic rule that fired on the invoked case would re-introduce a false positive on the never-invoked case. This is a false negative on an extraordinarily rare shape, consistent with the rule's stated *prefer-false-negatives-over-false-positives* bias; deliberately left unclosed (Codex review round 5).
- **P2 false negative, found 2026-07-06**: A single fluent chain on a parameter-typed builder (e.g. `builder.BindConfiguration(...).ValidateDataAnnotations()` where `builder` is a method parameter) is still tracked correctly, because the receiver/outer-chain walk (`AddReceiverInvocations`) does not depend on the receiver being a local. The gap is narrower: the same-block *split-statement* local-chain scanner (`AddSubsequentLocalInvocations`/`AddPreviousLocalInvocations`) requires `ILocalSymbol` (`semanticModel.GetSymbolInfo(memberAccess.Expression).Symbol is ILocalSymbol`), so when a parameter-typed builder's bind and validation calls are split across separate statements in the same block — a shape already supported for local variables — the parameter-typed version goes untracked and a missing `ValidateOnStart()` on that shape goes unreported. This is a real false negative, consistent with the rule's conservative bias. Empirically reproduced 2026-07-06; see Current Shortlist item 2.
- **Fixed 2026-07-06 (iteration 92) — forward false positive**: for a *local-variable* builder, `AddSubsequentLocalInvocations` previously `break`ed the forward scan on the first statement whose receiver was neither the tracked local nor an invocation, so a genuine `ValidateOnStart()` placed *after* an intervening unrelated statement was never collected (`var b = services.AddOptions<T>().BindConfiguration(...); b.ValidateDataAnnotations(); services.AddSingleton<X>(); b.ValidateOnStart();` collected `{BindConfiguration, ValidateDataAnnotations}` but not `ValidateOnStart`, so CFG003 wrongly fired). The scan now `continue`s past an inert intervening statement (an `ExpressionStatementSyntax` or `LocalDeclarationStatementSyntax`) and keeps collecting same-local calls, stopping (`break`) when a statement reassigns the tracked local or passes it by `ref`/`out` to a call that may reassign it (`StatementRetargetsLocal`, so a later `ValidateOnStart()` on a *different* builder bound to the same variable is not mis-attributed) and at any other statement kind — control flow such as `if`/`return`/`throw`/loops/switch, nested blocks, `using`/`lock` — that could keep the later call from executing on every path. Regression tests: `Cfg003_does_not_report_when_validate_on_start_follows_unrelated_statement` (the closed false positive), `Cfg003_reports_when_builder_local_reassigned_before_validate_on_start` (the reassignment guard), `Cfg003_reports_when_builder_local_passed_by_ref_before_validate_on_start` (the ref/out guard), and `Cfg003_reports_when_validate_on_start_is_behind_conditional_return` (the control-flow boundary). Because the split-statement scanner is shared with CFG004, a latent CFG004 forward false positive was corrected in the same change (`Cfg004_honors_later_local_bind_statement_validation_after_unrelated_statement`, formerly `Cfg004_stops_..._scan_at_unrelated_statement`, now asserts no diagnostic). Precision restored `4→5`.
- **Backward mirror (open, P2/low), characterized 2026-07-06 (iteration 92)**: the *prior* scan `AddPreviousLocalInvocations` still stops at the first unrelated statement, so the rarer, contrived shape where a validation call sits *before* the bind separated by an unrelated statement (`optionsBuilder.ValidateDataAnnotations(); Validate(optionsBuilder); optionsBuilder.BindConfiguration("Stripe");`) is dropped and CFG003/CFG004 still fire though validation is enabled at runtime. Pinned by `Cfg004_stops_later_local_bind_statement_prior_scan_at_unrelated_statement` and `Cfg004_does_not_scan_local_builder_initializer_across_unrelated_statement`. The backward scan's return value is the initializer-boundary index consumed by `AddLocalInitializerInvocations`, so making it symmetric requires reworking that boundary contract; deferred to a focused follow-up rather than widening the forward fix. This shape (validation registered before the bind, across an unrelated statement) is materially rarer than the forward pattern, so it does not on its own re-drop Precision below `5`.
- Future code-fix work should be driven by concrete formatter regressions rather than speculative chain shapes.
- Documentation explains the rule and the fluent-chain shapes it recognizes.

### CFG004 DataAnnotations Not Enabled

Reports when an options type has bindable DataAnnotations anywhere in its supported options graph but the options registration does not call `ValidateDataAnnotations()`.

Current behavior:

- Includes root, inherited (property-level), nested, and type-level validation attributes declared anywhere in the registered options type's base chain (inheritance-aware via `HasTypeLevelValidationInChain`, shared with `CFG002`'s required-key proof), on supported bindable option graphs, including single-constructor-bound properties inherited from base options types and initialized get-only object/collection properties, plus options types implementing `IValidatableObject` (inheritance-aware via `AllInterfaces`).
- Honors conservative bindable-property boundaries and only includes get-only properties when the binder can populate them through a mapped constructor parameter or an initialized nested object/collection instance.
- Includes public private-set properties only for registrations that explicitly set `BindNonPublicProperties = true` on the binder-options lambda parameter.
- Covers named and unnamed `OptionsBuilder<TOptions>` registrations when DataAnnotations validation stays on the same supported builder chain.
- Treats framework `Validate(...)` predicate registrations as validation for `CFG003`, but not as a substitute for `ValidateDataAnnotations()`.
- Tracks the framework `ValidateDataAnnotations()` before and after the binding call in the same fluent chain or immediate same-block local chain.
- Offers a fix that appends `ValidateDataAnnotations()` and appends `ValidateOnStart()` only when startup validation is missing, using the formatter-safe chain appender shared with `CFG003`.
- Does not append `ValidateOnStart()` when the registration started with `AddOptionsWithValidateOnStart<TOptions>()`.

Known gaps:

- Future code-fix work should be driven by concrete formatter regressions rather than speculative chain shapes.
- Does not infer annotations on non-bindable members, which is intentional but worth keeping explicit in docs.
- Fixed 2026-07-01 (see Improvement Changelog): a type-level (class) `ValidationAttribute` declared only on a base options class is now detected, because `ContainsValidationAttributes` reuses the same inheritance-aware `HasTypeLevelValidationInChain` helper `CFG002`'s required-key proof already relies on for this shape.
- Fixed 2026-07-06 (iteration 92, see Improvement Changelog): CFG004 shares the split-statement local-chain scanner (`AddSubsequentLocalInvocations`) with CFG003, so it inherited the same forward false positive — a `ValidateDataAnnotations()` genuinely called on a local builder after an intervening unrelated statement was dropped and CFG004 wrongly fired. The forward scan now skips intervening statements (stopping only at a reassignment of the tracked local), so that latent false positive is closed; the backward prior-scan mirror remains open (tracked under CFG003 Known Gaps). Precision was already `5` (the false positive had been mis-encoded as expected behavior), so this correction removes a real false positive without a score change.

### CFG005 Nested Validation Not Recursive

Reports when a nested object or collection item type contains validation attributes but the parent property is missing the required recursive validation attribute.

Current behavior:

- Finds nested object graphs and nested collection graphs that contain property-level DataAnnotations, type-level DataAnnotations declared anywhere in the nested type's base chain (shared `ContainsValidationAttributes`/`HasTypeLevelValidationInChain` fix with `CFG004`), or implement `IValidatableObject`.
- Treats single-constructor-bound nested objects, including inherited public properties populated through a derived constructor, and initialized get-only object/collection properties as bindable, matching runtime binder behavior.
- Includes public private-set nested properties only when `BindNonPublicProperties` is explicitly enabled on the binding call.
- Covers arrays, `IEnumerable<T>` shapes, nullable nested properties, and deep nested properties.
- Treats user namespaces such as `Systematic.Options` as analyzable user code while still excluding BCL `System` / `System.*` types.
- Suppresses interface-typed nested properties, dictionary value objects, and already annotated recursive-validation properties.
- Offers fixes for `[ValidateObjectMembers]` and `[ValidateEnumeratedItems]`, including options properties declared in a different source document from the registration diagnostic, inherited property declarations, constructor-bound object and collection record parameters that require a `property:` attribute target, files that use namespace-local using blocks, and files with same-named local attributes that require fully qualified attribute insertion.

Known gaps:

- More complex custom recursive validation patterns are intentionally not inferred.
- Fixed 2026-07-01 (see Improvement Changelog): a nested object/collection-item type whose only validation constraint is a type-level (class) `ValidationAttribute` declared only on a base class is now detected as needing recursive validation, since `GetNestedValidationCandidates` calls the same `ContainsValidationAttributes` helper as `CFG004`, which now reuses the inheritance-aware `HasTypeLevelValidationInChain` helper `CFG002` already relies on.
- **Fixed 2026-07-06 (iteration 95) — struct false negative**: `GetNestedValidationCandidates` gated nested candidates on the same `IsPotentialNestedObject` helper as `CFG006`, whose body was `type.TypeKind == TypeKind.Class`, so a nested struct- or record-struct-typed options property carrying validation attributes yielded no candidates and `CFG005` stayed silent even though the runtime binder populates struct properties and `Validator.TryValidateObject` would validate them. Fixed by extending `IsPotentialNestedObject` to also accept `TypeKind.Struct` (primitive/BCL value types stay excluded by the existing `SpecialType`/`System`-namespace guards; enums are `TypeKind.Enum`). Regression test: `Cfg005_reports_nested_struct_without_recursive_validation`. Same shared-helper change closed the `CFG006` struct gap.

### CFG006 Unknown Configuration Key

Reports an appsettings key under a bound section when the key does not match a bindable options property or its `[ConfigurationKeyName]` override. This rule stays informational because configuration binding is flexible and false-positive cost is higher.

Current behavior:

- Checks every matching bound section across visible `appsettings.json` and dot-qualified `appsettings.*.json` files for supported named and unnamed `BindConfiguration(...)`, `Bind(GetSection(...))`, `Bind(GetRequiredSection(...))`, and direct `Configure<T>(GetSection(...))` / `Configure<T>(GetRequiredSection(...))` registrations.
- Treats single-constructor-bound records/classes, including inherited public properties, and initialized get-only object, collection, and dictionary properties as bindable, while leaving uninitialized get-only properties outside the static bindable graph when no mapped constructor parameter exists.
- Honors constant-true `BindNonPublicProperties` assignments on the binder-options lambda parameter for public private-set options properties and stays on the default public-setter boundary otherwise.
- Parses `//` and `/* ... */` comments, JSON string escapes, and colon-delimited JSON keys before walking keys so commented local appsettings files stay analyzable.
- Merges sibling colon-delimited keys under the same nested object into one projected configuration node before recursive unknown-key analysis.
- Recurses into constructor-bound nested objects, nested object properties, arrays/lists of nested objects, strongly typed dictionary values, and dictionary values containing nested object collections.
- Honors `[ConfigurationKeyName]` overrides at the root and nested levels for property-bound options, matching the configured key instead of the CLR property name when an override is present. Constructor-bound properties use constructor parameter keys because the runtime binder requires those keys; aliases on settable constructor-bound properties, including private setters only when `BindNonPublicProperties` is enabled, are accepted only when the constructor key is also present in the same section or the matching constructor parameter has a default value.
- Treats user namespaces such as `Systematic.Options` as analyzable user code while still avoiding BCL `System` / `System.*` nested types.
- Treats scalar array items and dictionary entry names as values rather than property names.

Known gaps:

- Does not model provider precedence or environment-specific override intent.
- Should not become a warning unless real-world coverage proves the false-positive profile stays low.
- No automatic fix because the diagnostic is informational and anchored in JSON additional files; `CFG006` is intentionally excluded from `ConfigContrabandCodeFixProvider.FixableDiagnosticIds`.
- **Fixed 2026-07-06 (iteration 95)**: `IsPotentialNestedObject` filtered on `TypeKind.Class`, so struct- and record-struct-typed nested options properties were excluded from recursive unknown-key analysis even though the real `ConfigurationBinder` binds and recurses into them, so a typo under a struct-typed nested property (e.g. `App:Database:ConnetionString` where `Database` is a `struct DatabaseOptions`) went undetected. Fixed by extending `IsPotentialNestedObject` to also accept `TypeKind.Struct`; the same change closed the shared `CFG005` gap. Regression test: `Cfg006_reports_unknown_key_under_nested_struct_options_object`. The JSON schema builder keeps its own class-only nesting so unknown value types stay permissive in schema output (a struct may be `TypeConverter`-bound from a scalar), leaving the schema surface unchanged.

### CFG007 Unknown Configuration Key Will Throw

Reports strict-binding key failures related to the `CFG006` unknown-key model, but only when the supported binding call explicitly enables `BinderOptions.ErrorOnUnknownConfiguration`. In that mode the runtime binder throws instead of silently ignoring a rejected key, so the diagnostic is a Warning.

Current behavior:

- Covers `BindConfiguration(...)`, `Bind(GetSection(...))`, and direct `Configure<T>(GetSection(...))` registrations that pass a binder-options lambda.
- Mostly reuses the `CFG006` bindable-property graph, including nested objects and all existing constructor-bound, initialized get-only, dictionary, collection, and private-set boundaries.
- Reports `[ConfigurationKeyName]` alias keys that loose binding accepts but current strict binding rejects.
- Requires the final visible assignment to `ErrorOnUnknownConfiguration` on the actual binder-options lambda parameter to be constant `true`; reset-to-false, non-constant, compound writes, early-return, and nested control-flow assignments stay on the conservative path. Recognizes a tuple-deconstruction assignment, including nested tuples (e.g. `(options.ErrorOnUnknownConfiguration, options.BindNonPublicProperties) = (true, false);` or `((options.ErrorOnUnknownConfiguration, options.BindNonPublicProperties), unused) = ((true, false), 0);`), the same way as a direct member-access assignment, matching each left-side tuple element against the target property and reading the correspondingly-positioned right-side element's constant value; stays conservative when a sibling tuple element aliases the runtime `BinderOptions` object to another variable, reassigns the parameter itself, or passes it into a helper-call argument in the same statement, reusing the existing escape-analysis proof for the last case. Fixed 2026-07-06 (see Improvement Changelog).
- Treats a reset delegate/lambda that references the runtime binder options and is passed as a call *argument* (an inline lambda/anonymous method, or a local delegate variable — `RunNow(() => options.ErrorOnUnknownConfiguration = false)` or `RunNow(disableStrict)`) as an escape, staying conservative (`CFG006` Info) the same way a directly-invoked reset delegate (`disableStrict()`, `.Invoke()`, `?.Invoke()`) already does. Fixed 2026-07-06 (iteration 94, see Improvement Changelog).
- Reports object-shaped data under scalar properties, such as `"ApiKey": { "Foo": "x" }`, when strict binding treats the child key as an error; scalar CLR property names accepted by strict binding stay quiet unless they contain a deeper rejected child.
- Treats explicit `null`/`default` initializers on settable nested objects as declared-type binding, while keeping null CLR-only nullable values informational because the runtime does not descend into them.
- Reports rejected object-shaped entries inside scalar arrays/lists, scalar dictionary values, and nested dictionary scalar, object, or object-collection values, including string collection/dictionary child names such as `Length` that look like CLR members but still fail strict scalar-item conversion.
- Keeps open interface/object declared or value shapes, property- or constructor-initialized polymorphic reference shapes, and matching initializer- or constructor-prepopulated polymorphic dictionary entries, including ignore-case dictionary comparers, on the `CFG006` informational path because strict binding may bind an existing runtime subtype that exposes members not present on the declared base type.
- Keeps default binding, CLR property names that strict binding ignores or accepts without throwing, helper calls that receive the runtime `BinderOptions`, and unrelated `BinderOptions` instances on the existing `CFG006` informational path.

Known gaps:

- Does not infer helper methods, variables, or broader control flow that might enable or disable strict binding elsewhere.
- **Fixed 2026-07-06 (iteration 94) — false positive**: a reset delegate/lambda that sets `options.ErrorOnUnknownConfiguration = false` and is passed as a call *argument* to a helper that invokes it (e.g. `Action disableStrict = () => options.ErrorOnUnknownConfiguration = false; RunNow(disableStrict);`, or the inline `RunNow(() => options.ErrorOnUnknownConfiguration = false)`) previously escaped the strict-binding proof: `ContainsRuntimeBinderOptionsEscape`'s argument scan only matched a bare `IsRuntimeBinderOptionsReference` argument (not a captured delegate) and `InvocationMayRunLocalBinderOptionsHelper` only resolved *directly-invoked* delegates, so the final value stayed `true` and CFG007 escalated to a Warning even though the helper turns strict mode off before binding. Fixed by `ArgumentMayCaptureRuntimeBinderOptions`, which the argument scan now also consults: an inline lambda/anonymous-method argument (checked via `AnonymousFunctionReferencesRuntimeBinderOptions`) or a local-delegate-variable argument (checked via `LocalDelegateMayReferenceRuntimeBinderOptions`) whose body references the runtime binder options is treated as an escape, so the analyzer stays conservative and reports the softer `CFG006` Info instead. Regression tests: `Cfg006_stays_info_when_error_on_unknown_configuration_is_reset_through_delegate_passed_as_argument` and `..._through_inline_lambda_argument`. Consistent with the already-handled directly-invoked reset-delegate path and the rule's prefer-false-negatives bias.
- Intentionally does not model order-sensitive tuple-deconstruction cases where one sibling element writes the target property on the original binder-options object while another sibling element simultaneously reassigns the parameter itself to a new instance (e.g. `(options.ErrorOnUnknownConfiguration, options) = (true, new BinderOptions());`). C# evaluates the property write against the original object before the parameter is repointed, so the true runtime state can be `true` even though the analyzer conservatively treats the whole tuple as inconclusive (a false negative, not a false positive) — reviewed and accepted 2026-07-06 given the shape's rarity and the added complexity a precise order-sensitive fix would require.
- Does not offer a code fix because the diagnostic is anchored in an appsettings additional file.
